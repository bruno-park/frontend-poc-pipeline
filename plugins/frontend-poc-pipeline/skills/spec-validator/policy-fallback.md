## 1. 목적

화면기획서의 디스크립션 작성 방식의 표준을 정의한다.

- 기획서 가독성 향상
- 피그마 ↔ 구글 스프레드시트 동기화 지원
- AI 기반 기획서 분석 및 자동화 가능 구조 확보
- 기획서 간 작성 방식 통일

---

## 2. 디스크립션 기본 구조

디스크립션은 아래 **4개 컬럼 구조**로 작성한다.

| 컬럼 | 설명 |
|------|------|
| no | 화면 내 숫자 태그와 매칭되는 번호 |
| title | 해당 번호가 가리키는 UI 영역 또는 요소 |
| setup | 기본 상태 (초기값, 기본 정렬, 최초 진입 상태 등) |
| rules | 상세 규칙 |

---

## 3. 각 컬럼 작성 기준

### 3.1 no

- 화면 설계 화면에 표시된 **번호 태그와 동일하게 작성**
- 세부 번호 체계 유지 가능 (예: 3-1, 6-2)

### 3.2 title

- 가능한 한 **명확한 영역 이름 사용**
- UI 요소 또는 기능 중심으로 작성
- 예: 검색 영역, 광고상품 목록, 다운로드 버튼

### 3.3 setup (기본 상태)

포함 내용:
- 최초 진입 상태
- 기본 정렬 기준
- 기본 선택값
- 초기 노출 여부
- 기본 필터 상태

작성 규칙:
- **태그([F][P][Data]) 사용 금지**
- 내용이 없으면 `-` 입력

예: `기본 정렬: 등록일 내림차순`, `최초 진입 시 전체 데이터 조회`

---

## 4. rules 작성 규칙

rules는 **태그 기반 구조**로 작성한다.

| 태그 | 의미 |
|------|------|
| [F] | 사용자 액션 및 화면 동작 |
| [P] | 정책 및 처리 규칙, 단순 텍스트 및 문구 |
| [Data] | 데이터 정의 |

---

## 5. rules 태그 상세 정의

### 5.1 [F] (Flow / Interaction)

**"만약 ~하면, ~된다"** 와 같은 사용자 행동에 따른 **UI 동작**을 설명한다.

- **작성법:** `'조건' -> '결과'` 형태로 작성

예:
```
[F] 클릭 시 상세 팝업 노출
[F] '일부' 선택 시 남성/여성 체크박스 노출
[F] 'PC'만 선택 시 하단 영역 딤드(Dimmed) 처리
```

**페이지(화면) 또는 메뉴 이동 규칙:**

형식: `→ '페이지명' 페이지(화면) 또는 메뉴로 이동`

예:
```
[F] 게재위치명 클릭 시 → '게재위치 상세 조회' 페이지로 이동
[F] 광고상품명 클릭 시 → '광고상품 조회' 페이지로 이동 (현재 탭)
[F] 다운로드 버튼 클릭 시 → '다운로드 이력' 페이지로 이동 (새 탭)
```

### 5.2 [P] (Policy)

**"~해야 한다"** 와 같은 로직, 처리 규칙과 **"현황"** 을 서술하기 위한 단순 텍스트, 영역 설명, 선택값 옵션 등을 설명한다.

예:
```
[P] 광고상품명은 매체 내 중복 불가
[P] 최근 90일 데이터만 조회 가능
[P] 운영 보고서 타이틀 및 우측 설명 문구 표시
[P] 리스트: 전체, ON, OFF
```

### 5.3 [Data] (데이터 정의)

테이블의 **컬럼(필드) 정의가 필요한 경우에만** 작성한다.

단순 값 나열, 레이블, 선택 옵션 등은 포함하지 않는다.

작성 규칙:
- `[Data]`는 **블록 구조로 작성**
- 태그는 **한 번만 선언**
- 하위 필드는 `-` 사용

예:
```
[Data]
- 광고상품명: 광고상품 이름
- 등록일: 상품 등록 날짜
- 상태: 운영 상태
```

## 6. rules 작성 순서

```
[F] → [P] → [Data]
```

단, **사용자 흐름상 자연스러운 경우 일부 순서 변경은 허용한다.**

## 7. 밸리데이션(Validation) 및 메시지 정책

### 7.1 밸리데이션(Validation) 정책

- **ID 체계:** `v_{category}_{rule}` 형태
- **Figma 작성:** 디스크립션 본문에는 문구를 쓰지 않고, 표 우측에 **[밸리데이션 전용 표]** 배치
  - 표 구성 항목: `Field` / `Case` / `Validation_ID` / `Param`
- **본문 기재 예외:** 해당 메뉴/기능에만 필요한 특수한 정책이 있는 경우, Params 정책이 필요한 경우

### 7.2 시스템(얼럿) 메시지 관리 원칙

- **공통 메시지:** 삭제 확인, 저장 완료, 권한 없음 등 → **DS(Design System) 컴포넌트**에서 관리
- **특화 메시지:** 해당 기능에만 국한된 안내 문구 → **Figma 화면 본문 및 디스크립션**에 직접 기술

### 7.3 요약

| 구분 | 대상 | 관리 주체 | 기획서 작성 위치 |
|------|------|-----------|-----------------|
| Validation | 필수값, 형식 오류 등 입력값 검증 | 마스터 파일 (ID 기반) | 디스크립션 우측 별도 표 |
| 시스템 메시지 | 삭제/성공 얼럿, 액션 불가 안내 등 | DS(공통) / 기획서(특화) | Figma 화면 및 디스크립션 본문 |

## 8. 피그마 작성 가이드 (실무 Tip)

1. **Rule 구분이 헷갈릴 때:** 유저 액션에 따른 변화는 **[F]**, 시스템의 약속(제약/계산)은 **[P]**
   - 구분이 모호하면 불렛 포인트(•)로 나열 후 AI에게 분류 요청
2. **조건부 UI 변경 작성 위치:** 영향을 주는 요소 기준으로 작성
   - case: A값 체크박스 선택 → C 필드 활성화 → A가 넘버링된 디스크립션에 작성
3. **TBD 사용:** 정책이 확정되지 않은 경우
   - `[TBD] 업로드 파일 최대 용량 협의 필요`
